Office hours cheat sheet

Meeting with Dr. Sharif — SafeGuard: Prompt injection interception MCP for AI coding agents

Opening pitch (30 sec): We're building an MCP server that sits between AI coding agents and the web. When an agent fetches a URL, our server intercepts it, scans the HTML for prompt injections and malicious content, and either passes it through or alerts the developer. First milestone: URL interception only. Think of it as an application-layer firewall for AI agents.
1
Appropriate defenses

We use a defense-in-depth approach with three layers, not just "ask an LLM if it's safe":

Layer 1 — Threat intelligence (pre-fetch)

Check the URL against known-malicious databases before even downloading the page. Instant block if the domain is flagged.

URLhausGoogle Safe BrowsingPhishTankVirusTotal
Layer 2 — Structural analysis (post-fetch)

Parse the HTML with BeautifulSoup and flag content that is intentionally hidden from humans but readable by machines. The mere presence of hidden text is a red flag.

display:none divszero-font textwhite-on-whitehidden iframesobfuscated scriptsmetadata injections
Layer 3 — Semantic analysis (LLM-as-judge)

Pass the cleaned text to a fast, dedicated screening model (Claude Haiku or Bedrock Guardrails) with one job: "Is this text an instruction directed at an AI agent, or normal content meant for humans?" Returns a structured risk score.

Claude Haiku APIAmazon Bedrock Guardrails
Key point: Each layer catches different things. Layer 1 catches known-bad URLs cheaply. Layer 2 catches hiding techniques regardless of content. Layer 3 catches novel injections that don't match known patterns. We can measure each layer's contribution independently.
2
Specific prior work

Our project builds directly on established research in three areas:

Foundational attack research

Greshake et al. (2023) — "Not what you've signed up for" — Formalized indirect prompt injection: attackers embed instructions in external content (web pages, emails, documents) that LLMs process. Published at AISec '23. This is the core threat model for our project.

Aichberger et al. (2025) [ref 1 in course doc] — Attacking multimodal OS agents with malicious image patches.

Li et al. (2025) [ref 9 in course doc] — Commercial LLM agents are already vulnerable to simple yet dangerous attacks.

Benchmarks we'll evaluate against

BIPIA (Microsoft, Yi et al. 2023/2025) — First benchmark for indirect prompt injection attacks. 5 application scenarios, 250 attacker goals. Published at KDD '25. All 25 tested LLMs were found vulnerable.

LLMail-Inject (SaTML 2025) — 208K+ unique attack prompts in a realistic email assistant scenario.

PINT Benchmark (Lakera) — 4,314 labeled inputs across English and non-English.

Existing defense frameworks (inspiration)

Lakera Guard — Commercial real-time prompt injection API. 98%+ detection, sub-50ms latency. Acquired by Check Point in 2025.

NVIDIA NeMo Guardrails — Open-source programmable conversation rails for LLM apps.

Amazon Bedrock Guardrails — Native prompt attack detection filters on AWS.

Framing for Dr. Sharif: Lakera/NeMo/Bedrock protect LLM apps at the API level. Our contribution is different: we protect AI agents at the tool level via MCP — intercepting content before it enters the agent's context window. This is a "shift-left" approach.
3
System architecture

Hybrid cloud-local architecture. The local MCP server handles agent communication and the human-in-the-loop UI. The cloud backend (AWS) handles the dangerous fetch and all scanning in isolation.

Agent calls safe_fetch(url) → Local MCP server forwards URL to AWS → AWS Lambda fetches HTML in isolation → Layer 1: Threat intel check (URLhaus, Safe Browsing) → Layer 2: BeautifulSoup strips hidden content → Layer 3: Bedrock / Haiku semantic scan → Risk assessment returned to local MCP → Low risk: clean text passed to agent silently → High risk: terminal alert → user approves/blocks

Why hybrid: Fetching untrusted URLs locally risks SSRF or sandbox escapes. Cloud isolation means malicious payloads never touch the developer's machine. But the human-in-the-loop UI stays local for instant, frictionless alerts.

Budget: ~$150 total. EC2 t3.small (~$35/4mo) + Claude API for semantic scanning (~$20) + free threat intel APIs. Plenty of headroom.

If he asks about enforcement: System prompt instructions are our first layer (tell Claude Code to always use safe_fetch). For stronger enforcement, we can set HTTP_PROXY/HTTPS_PROXY to route all agent traffic through a local mitmproxy that forwards to our scanner. We'll measure the gap between soft and hard enforcement in our evaluation.
4
Usability aspects

This is the workshop angle — we're not just building security, we're building usable security.

Preventing alert fatigue

Clean URLs load instantly with zero friction — the developer doesn't even know the scan happened. Only genuinely suspicious content triggers an interruption. Configurable risk thresholds let developers tune sensitivity.

Actionable terminal alerts

When something is flagged, the alert shows exactly what was found, where in the page, and why it's suspicious — not a generic "blocked" message. The developer can approve, block, or open a full scan report with one keystroke.

Transparency via scan reports

A localhost web dashboard shows the full document with highlighted suspicious segments, risk breakdown by detection layer, and a timeline of the scan. Developers can verify the tool's decisions and build trust.

User study design

Within-subjects study: participants use a coding agent on tasks involving injected web content. Condition A: no protection. Condition B: SafeGuard active. Measure detection accuracy, response time, false positive tolerance, and SUS usability score.

Key phrase to use: "Positive-sum design" — the tool improves security without degrading the development workflow. Clean sites have near-zero latency overhead. The developer only pays attention cost when there's a real threat.

Closing ask

We're focusing strictly on URL/HTML interception for the first milestone to keep the MVP achievable. Extensions to PDFs, documents, and files are planned if time permits. Does this scope feel right, and do you have any suggestions on the evaluation methodology or the defense layers?
